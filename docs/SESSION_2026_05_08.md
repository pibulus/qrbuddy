# Session 2026-05-08: File Locker Recovery

## What Broke

File lockers were live but scanner uploads were effectively blocked. The bucket
QR sent people to `/bucket/{code}`, but `upload-to-bucket` required an
`owner_token`. Only the creator had that token, so guests could open the locker
page but could not fill it.

There were two production wiring issues on top:

- `qrbuddy.app` was still routed through a Cloudflare Tunnel to another host's
  `localhost:8004`, which served an older Fresh bundle.
- Deno Deploy had stale Supabase env vars for a different Supabase project.

## Fixes

- Allowed public uploads to empty lockers when the scanner has the bucket URL.
- Kept owner token support for creator control.
- Required PIN/password for protected lockers before public upload or download.
- Fixed text/link locker downloads by returning `content_data` from
  `claim_bucket_download`.
- Added `X-Bucket-Emptied` headers for text/link downloads so the UI clears
  after destructive downloads.
- Sanitized download filenames.
- Changed locker creation throttling from a permanent 3-per-IP cap to a daily
  cap.
- Moved mobile Power-Ups and Templates menus to compact grids with settings
  below the grid.

## Production Notes

- Apex DNS for `qrbuddy.app` now points at Deno Deploy via `qrbuddy.deno.dev`.
- `www.qrbuddy.app` was left on the existing tunnel because Deno Deploy does not
  currently list it as a custom domain.
- Supabase secret `APP_URL` was set to `https://qrbuddy.app`.
- Live Supabase project is `aqydpibnvlhcjcwosrti`.
