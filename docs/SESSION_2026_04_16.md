# What We Fixed (and Why It Was Broken)

**A play-by-play of the QRBuddy sanity pass, April 2026**

## The Problem

You opened qrbuddy.app, tried the File Locker feature, and got this in the console:

```
Access to fetch at 'https://aqydpibnvlhcjcwosrti.supabase.co/functions/v1/create-bucket'
from origin 'https://qrbuddy.app' has been blocked by CORS policy
```

That error is your browser being a bodyguard. Let's break down what was happening.

## Part 1: The CORS Fix

### What is CORS, actually?

When your browser loads `qrbuddy.app` and the JavaScript tries to talk to
`aqydpibnvlhcjcwosrti.supabase.co` (a different domain), the browser gets
suspicious. It's like calling a restaurant and they ask "who gave you this
number?"

Before your actual request goes through, the browser sends a **preflight
request** — an `OPTIONS` request that basically asks:

> "Hey supabase.co, is it cool if qrbuddy.app talks to you?"

The server needs to respond with a header that says:

```
Access-Control-Allow-Origin: https://qrbuddy.app
```

If it says anything else (or nothing), the browser blocks the request entirely.
Your JavaScript never even sees it fail — it's blocked at the network level.

### What was wrong

Every edge function imported CORS headers from a shared file:

```
supabase/functions/_shared/cors.ts
```

That file had this logic:

```typescript
function getAllowedOrigin(): string {
  const isProduction = Deno.env.get("DENO_DEPLOYMENT_ID");
  if (isProduction) {
    return "https://qrbuddy.deno.dev";  // <-- THE BUG
  }
  return "http://localhost:8000";
}
```

It was hardcoded to `qrbuddy.deno.dev`. But the app was being served from
`qrbuddy.app`. The browser sent `Origin: https://qrbuddy.app`, the server
responded with `Access-Control-Allow-Origin: https://qrbuddy.deno.dev`, and the
browser said **nope**.

### The fix

We replaced the single hardcoded origin with a **whitelist** that checks the
incoming request:

```typescript
const ALLOWED_ORIGINS = [
  "https://qrbuddy.app",
  "https://qrbuddy.deno.dev",
  "http://localhost:8000",
  "http://localhost:8004",
];

export function getCorsHeaders(request?: Request) {
  const origin = request?.headers.get("Origin");
  if (origin && ALLOWED_ORIGINS.includes(origin)) {
    return {
      "Access-Control-Allow-Origin": origin,  // Echo it back
      // ...
    };
  }
}
```

The key insight: **the server echoes back the origin it matched**. If qrbuddy.app
asks, it gets `qrbuddy.app` back. If qrbuddy.deno.dev asks, it gets that back.
The browser sees its own origin reflected and is happy.

### But wait — every function needed updating

The old code used a **static** `corsHeaders` object that was created once when
the module loaded (no request available yet). Every function did:

```typescript
import { corsHeaders } from "../_shared/cors.ts";

// In every response:
headers: { ...corsHeaders, "Content-Type": "application/json" }
```

We changed all 13 edge functions to use the **dynamic** version that takes the
request:

```typescript
import { getCorsHeaders, createCorsResponse } from "../_shared/cors.ts";

// OPTIONS preflight:
if (req.method === "OPTIONS") {
  return createCorsResponse(req);  // <-- passes req through
}

// In every response:
headers: { ...getCorsHeaders(req), "Content-Type": "application/json" }
```

This was 70+ individual edits across 13 files. We used subagents (parallel
workers) to do 4 batches simultaneously.

### How we verified it

```bash
curl -X OPTIONS \
  "https://aqydpibnvlhcjcwosrti.supabase.co/functions/v1/create-bucket" \
  -H "Origin: https://qrbuddy.app" \
  -H "Access-Control-Request-Method: POST" \
  -D -
```

This simulates exactly what the browser does. The `-D -` flag prints the response
headers. We saw:

```
access-control-allow-origin: https://qrbuddy.app
```

CORS fixed.

## Part 2: The Edge Functions Weren't Deployed

Here's the twist — fixing the code was only half the problem. When we checked
what was actually running on Supabase:

```bash
supabase functions list --project-ref aqydpibnvlhcjcwosrti
```

```
ID | NAME | SLUG | STATUS | VERSION
---|------|------|--------|--------
                                      <-- EMPTY
```

**Zero functions deployed.** The project `aqydpibnvlhcjcwosrti` was created
(April 9) but nobody had pushed the edge functions to it. Every API call from
the browser was hitting a URL that didn't exist.

### Deploying edge functions

Supabase edge functions are Deno scripts that run on Supabase's servers. You
deploy them with the Supabase CLI:

```bash
supabase functions deploy create-bucket \
  --project-ref aqydpibnvlhcjcwosrti \
  --no-verify-jwt
```

What this does:
- `functions deploy create-bucket` — takes the code from
  `supabase/functions/create-bucket/index.ts`
- `--project-ref` — which Supabase project to deploy to
- `--no-verify-jwt` — don't require a signed JWT (we use the anon key as a
  Bearer token instead)

The CLI is smart enough to bundle any `../_shared/` imports automatically. We
deployed all 13 functions.

### The database was also incomplete

The tables existed (`file_buckets`, `destructible_files`, `dynamic_qr_codes`)
but several columns and functions were missing. The migrations had never been
run. We checked with:

```bash
supabase db query --linked \
  "SELECT column_name FROM information_schema.columns
   WHERE table_name = 'file_buckets';"
```

Missing: `download_started_at`, `delete_on_download`, `creator_ip`,
`last_accessed_at`. Plus no `scan_logs` table and no `claim_bucket_download` RPC
function.

We wrote all the missing SQL into one file and applied it:

```bash
supabase db query --linked -f /tmp/qrbuddy-migrations.sql
```

## Part 3: The Race Condition

This one's subtle. The dynamic QR redirect function had this flow:

```
1. Fetch QR from database (scan_count = 5)
2. Use scan_count to pick which URL to redirect to
3. Update scan_count to 6
```

The problem: if two people scan the same QR at the exact same time, both requests
read `scan_count = 5`, both redirect to URL #5, and both try to set it to 6. One
increment gets lost.

For sequential routing (where scan 1 goes to URL A, scan 2 goes to URL B, etc.),
this means people get the wrong URL.

### The fix: atomic operations

We created a PostgreSQL function that does the increment **inside the database**
in a single atomic operation:

```sql
CREATE FUNCTION increment_scan_count(p_short_code TEXT)
RETURNS INTEGER AS $$
DECLARE v_new_count INTEGER;
BEGIN
  UPDATE dynamic_qr_codes
  SET scan_count = scan_count + 1
  WHERE short_code = p_short_code
  RETURNING scan_count INTO v_new_count;
  RETURN v_new_count;
END;
$$ LANGUAGE plpgsql;
```

The magic is `scan_count = scan_count + 1` inside a single UPDATE statement.
PostgreSQL guarantees this is atomic — two concurrent updates will never produce
the same result. The `RETURNING` clause gives us the new value without a second
query.

In the edge function, instead of:

```typescript
// OLD: read-modify-write (race condition)
const count = qr.scan_count;
await supabase.update({ scan_count: count + 1 });
```

We now do:

```typescript
// NEW: atomic increment
const { data: newCount } = await supabase
  .rpc("increment_scan_count", { p_short_code: shortCode });
const scanIndex = newCount - 1;  // 0-based for array lookup
```

## Part 4: The Fresh App Deployment

QRBuddy has **two separate deployments**:

1. **The Fresh app** (the website itself) → lives on **Deno Deploy**
2. **The edge functions** (API endpoints) → live on **Supabase**

They're completely independent. Fixing edge functions doesn't update the website,
and vice versa.

The Fresh app is deployed with:

```bash
deployctl deploy --project=qrbuddy --prod --token=$DENO_DEPLOY_TOKEN
```

This pushes your local code to Deno Deploy's edge network. The app is then served
from data centers worldwide. The `--prod` flag makes it the live production
deployment (vs. a preview).

**You don't need the Pi for this.** The `deployctl` command runs from whatever
machine has the code and the deploy token. We ran it from your Mac. The app lives
on Deno's servers, not on the Pi.
